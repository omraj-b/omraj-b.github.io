When you see the 🔒 icon in your browser, something remarkable happens behind the scenes.

In a fraction of a second, your browser:

- verifies the identity of a server it has never met
    
- securely agrees on encryption methods
    
- establishes a shared secret
    
- and begins fully encrypted communication.
    

The process that makes this possible is the **TLS handshake**.

---

## 🧠 The Core Problem TLS Solves

Before diving into steps, understand the real challenge that two unknown devices (trying to talk to each other over a insecure network) face:

```text
How do two strangers on the internet:
1. Trust each other?
2. Share a secret securely? (This is for encryption)
3. Communicate efficiently afterward?
```

TLS is not just encryption.

```text
TLS = Trust + Key Exchange + Secure Communication
```

---

## 🧩 Big Picture Flow

```text
Client (Browser)                Server (Website)
------------------------------------------------
Hello →                         ← Hello + Certificate
Verify Identity
Establish Shared Secret
Secure Communication Begins →   ← App Data
```

---

## 🔄 Step 1: Client Hello (Solving Compatibility)

```text
Client ──Client Hello──▶ Server
```

"Client Hello" is a data packet sent by the client to the server it is trying to talk to. This data packet includes:

- Supported TLS versions
    
- Cipher suites
    
- Client Random
    
- Key Share (for modern TLS)
    

### 🧪 Wireshark View

![Client Hello](/assets/blog/HTTP_TLS/Client_hello.png)

Here the client is saying:

> “Here’s everything I support—choose what works best.”

### 💡 Why This Step Exists

```text
Different systems support different encryption methods. So, client provides info about this and hence the server only chooses what seems best and compatible with the client.
```

This step ensures:

- compatibility
    
- flexibility
    
- and forward security.
    

---

## 🔄 Step 2: Server Hello (Solving Agreement)

```text
Client ◀──Server Hello── Server
```

"Server Hello" is a data packet sent by the server to the client to confirm its choices. This data packet includes:

- Selected cipher suite
    
- Server Random
    
- Key Share
    

### 🧪 Wireshark View

![Server Hello](/assets/blog/HTTP_TLS/Server_hello.png)

Here server chooses and sends back the decided agreement about the encryption methods:

```text
Client proposes → Server chooses
```

Now both sides agree on:

- how encryption will work
    
- how keys will be generated
    

### 💡 Why This Step Exists

```text
Security not only depends on providing good choices but also in forcing those choices.

Although the client may provide several different options, it is up to the server to decide which one it will support due to its own security policies, ensuring that no outdated encryption techniques are used.

Here, the parameters of the communication protocol are effectively established before anything is said.
```

---

> Now that both sides agree on how to communicate, the next problem is trust. Trust is estabilished with the help of digital certificates.

---

## 🔐 Step 3: Digital Certificate (Solving Identity)

```text
Server ──Certificate──▶ Client
```

Contains:

- Public key
    
- Domain identity
    
- CA (Certificate Authority) signature
    

### ⚠️ Why You Don’t See It in Wireshark (TLS 1.3)

In TLS 1.3:

```text
Server Hello → Encrypted Handshake → Certificate
 (Contains)
```

The certificate is still sent but encrypted.

Here the server is saying:

> “This is my identity, verified by a trusted authority.”

### 💡 Why This Step Exists

```text
Encryption is useless without trust.
```

### 🌍 What if this step is skipped? Ex- Public Wi-Fi Attack

```text
Imagine connecting to public Wi-Fi at a café.

An attacker could intercept your traffic and pretend to be the website you’re visiting.

Without certificate verification, your browser would have no way to detect this.

TLS prevents this by verifying the server’s identity using trusted Certificate Authorities.
```

---

> Once trust is established, the next challenge is to securely share a secret that will be used to encrypt communication over an otherwise insecure network.

---

## 🔑 Step 4: Key Exchange (Solving Secret Sharing)

```text
Client (key share) ─────▶
                        Shared Secret
Server (key share) ◀────
```

This is done with the help of ECDHE algorithm.

### 🧪 Wireshark Reality

There is no separate packet. In TLS 1.3, key exchange does not occur in a separate step. Key exchange happens inside:

- Client Hello → key_share
    
- Server Hello → key_share
    

This exchanged key is used to derive the common symmetric encryption key for this session.

```text
Shared Secret = Derived, not transmitted (Using ECDHE method)
```

### 💡 Why This Step Exists

```text
Sending a secret directly is insecure.
```

---

## 🔒 Step 5: Secure Communication (The Goal)

```text
Client ◀══════ Encrypted Data ══════▶ Server
```

Our goal in this step is to encrypt the data using the secret key so that no one can read it.

### 🧪 Wireshark View

![Encrypted Data](/assets/blog/HTTP_TLS/Encrypted_app_data.png)

### 🔗 What’s Happening

```text
Readable structure → Encrypted payload
```

### 💡 Why This Step Exists

```text
Everything before this was setup.
This is the actual goal.
```

### 🌍 Example: Protecting Your Login

```text
When you log into a website, your username and password are sent as part of HTTP requests.

Without HTTPS, anyone on the network could read these credentials.

The 'S' in HTTPS comes from TLS, which makes the same request encrypted, making it unreadable to attackers.
```

Before ending and coming to conclusion I would like to add few points about:

- 🔄 **A Note on TLS Version Compatibility**  
      
    If you look closely at Wireshark, you might notice something confusing: even when TLS 1.3 is clearly in use, the record layer still shows TLS 1.2.  
      
    This is not a mistake—**it is by design**.  
      
    TLS 1.3 reuses the TLS 1.2 record format for backward compatibility, ensuring that newer implementations can still communicate across older network infrastructure. The actual version negotiation happens inside extensions like `supported_versions`, not in the record header itself.  
      
    In simple terms:  
    The label may say TLS 1.2, but the behavior and security are fully TLS 1.3. This subtle design choice allows the internet to evolve securely without breaking existing systems.
    

- **🔑 A Note on ECDHE (Elliptic Curve Diffie-Hellman)**  
      
    Modern TLS uses ECDHE for key exchange—a method that allows two parties to create a shared secret without ever sending that secret over the network. Instead, both the client and server exchange temporary public values (key shares) and independently compute the same secret using elliptic curve mathematics.  
      
    What makes ECDHE powerful? - **Forward secrecy**  
    Even if an attacker records encrypted traffic today and later compromises the server’s private key, they still cannot decrypt past sessions. This is because each connection generates a fresh, one-time key.  
      
    So to conclude:
    ECDHE ensures that secrets are never transmitted—and even if something is compromised later, past communications remain secure.

## 🧠 Final Insight

TLS is often misunderstood as “just encryption.” It is not.

```text
TLS is a system that:
- establishes trust
- negotiates security
- shares secrets safely
- and only then enables encryption
```

---

## Closing Thought

The TLS handshake makes it possible for two unknown machines to trust each other, communicate in a secure manner, and exchange their secrets without disclosing them to the rest of the world.

The steps captured by Wireshark are only part of that process; the remaining part takes place using cryptography under the hood.

So the next time you see the 🔒 icon, remember:

It represents a complete system that transforms an insecure network into a trusted communication channel—every single time you connect.
